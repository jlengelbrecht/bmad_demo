What is BMAD?

Lesson 1 · 12 min read

BMAD is a methodology for shipping AI-assisted code as a team. It's not a tool. It's not a framework. It's a set of rituals that make the shape of a story — what it asks for, who reviews it, when it's done — concrete enough that any teammate can pick it up and run it through any AI tool they happen to use.

The artifact chain

Every BMAD project produces six artifacts in order: a brief, a PRD, an architecture document, a list of epics + stories, an ADR, and the working code that implements story 1.1. Each artifact loads the previous one as context. Files are the contract.

You'll experience this directly in the capstone — chatting with your AI tool through each phase, watching files appear in your chosen directory as the agent works.

Why files, not chat history

The agent re-reads the brief at the start of every PRD turn. Re-reads the brief + PRD at the start of every architecture turn. Memory is on disk, not in conversation. This is what makes the contract portable across tools: claude-code, codex, and copilot all read the same brief.md.

▶ reading brief.md (1.2 KB)
Got it — your brief outlines a customer-onboarding redesign focused on reducing time-to-first-success for new accounts. Let me confirm the load-bearing facts before drafting the PRD:

From the brief:
• Customer: small-business admins setting up their first integration
• Problem: 60% drop-off at the integration-test step
• Success metric: median time-to-first-success under 12 minutes

Two questions before I write the PRD draft:
1. Are there constraints from the existing onboarding code we should respect, or is this a greenfield rewrite?
2. What's the scope tier — MVP only, or are we including v1.1 candidates?
