# Product Brief: BMAD Training Portal (`bmad_demo`)

## Executive Summary

`bmad_demo` is a runnable, self-referential **BMAD Training Portal** — a Next.js + Express + SQLite app distributed as a clonable Git repository — that teaches engineering teams how to adopt BMAD *as a team* and how to govern AI-assisted contributions in a shared codebase using GitHub-native controls (CODEOWNERS, branch protection, required reviews, lead-approval merge gates). **No new platform to procure, no migration, no data leaves the org.**

Its defining property is **self-reference**: the lessons inside the portal point trainees at *this* repo's own BMAD artifacts — its product brief, PRD, architecture, stories, and CODEOWNERS file — as the lesson material. Trainees experience a complete BMAD repository from inside a guided tour of itself. Think Exercism, but for a methodology rather than a syntax.

**The portal is already validated by a committed internal POC**: one product team has been authorized to adopt it as their first BMAD rollout. The portal exists to head off the failure mode every team hits when introducing AI agents — individuals move fast, artifacts drift, and code lands in `main` without the team alignment that keeps a codebase coherent. With 78% of dev teams using AI assistants but only 9% of enterprises rated "Ready" on AI governance maturity (Deloitte 2025), the timing for a team-adoption-and-governance playbook — packaged as a working repo, not a slide deck — is now.

## The Problem

Engineering organizations are adopting AI coding assistants faster than they can govern them. Five concrete pain points:

- **Artifact drift.** Individual engineers produce inconsistent specs, stories, and PRDs because nothing on the team makes consistency a precondition for merging.
- **Vibe-coded contributions land unreviewed.** Code generated by AI assistants reaches `main` without the alignment a coherent codebase requires. ACM's April 2026 TechBrief explicitly warns that vibe coding skips core engineering practices for security, reliability, and maintainability.
- **Mixed-tool teams have no shared protocol.** When teammates use different agentic tools (Claude Code, Copilot, Codex, OpenCode), the *implementation contract* between them gets lost.
- **Non-engineering contributors are excluded.** PMs, designers, and less-code-savvy collaborators *can* contribute under AI assistance — 63% of "vibe coders" today are non-developers — but teams have no governance pattern that lets them do it safely.
- **BMAD docs and SDD tools don't address the team layer.** Official BMAD curriculum and tools like GitHub Spec Kit teach the framework and the artifacts; neither prescribes how a real team adopts them, runs the rituals, or gates the merge.

The cost of the status quo is the recognized open problem in spec-driven development: **implementation faithfulness** — beautiful planning artifacts whose produced code drifts from spec intent because no human gate catches it.

## The Solution

A complete, runnable BMAD repository that *teaches itself*.

A trainee runs `git clone` and `npm run dev`, lands on `localhost:3000`, and walks through six lessons (~15 min each), three hands-on labs (solo, full-team, async-team), and a 60–90 minute capstone in which they run a full BMAD planning cycle end-to-end. Lesson prose tells the trainee to open specific files in their editor — `_bmad-output/planning-artifacts/architecture.md`, `.github/CODEOWNERS`, a real story file — so the curriculum is teaching the *artifact*, not just the lesson page.

Three audience paths share the same repo:

- **Engineer/Team-Trainee** (~3h self-paced or one half-day workshop): the full curriculum.
- **Stakeholder** (15 minutes): a focused demo script for engineering and product leadership evaluating BMAD adoption.
- **Facilitator**: a guide for engineering leads running internal half-day workshops.

The load-bearing abstraction is the **story file as a tool-agnostic contract** — a self-contained brief any teammate (or any AI agent) can implement without losing context. Lesson 5 makes mixed-tooling explicit: a team where one engineer uses Claude Code, another uses Copilot, and a third uses Codex collaborates coherently because they all consume and produce the same story shape.

The governance answer is **GitHub's native machinery**: CODEOWNERS routes review to the right humans; branch protection enforces required reviews; the lead engineer is the final-say merge gate. The portal does not claim CODEOWNERS itself prevents drift — it claims that *routing the right human to the gate, plus teaching that human what to look for*, prevents drift. Lesson 4 teaches the look-for. Lesson 5 teaches the rituals around the gate.

To keep the lesson alive past the capstone, trainees leave with a **one-page team-rituals checklist** they pin in their own repo — a reinforcement artifact that survives when the portal is closed.

## What Makes This Different

This is not another BMAD reference manual and not an SDD tool. The official BMAD project (46k+ stars) covers the framework. GitHub Spec Kit (Feb 2026) covers spec/plan/task primitives across 30+ agents. **The unowned wedge is team adoption plus governance, modeled by a working repo you can clone.** Specifically:

- **Self-reference as pedagogy.** Other curricula tell you about BMAD; this one *is* a BMAD repo and points trainees at its own artifacts. Exercism for a methodology, not a syntax.
- **GitHub-native governance, zero new tooling.** No procurement, no migration, no platform risk. Uses controls every GitHub-using team already has.
- **Mixed-tooling is a first-class lesson, not a footnote.** Story-as-contract is engineered into the curriculum so heterogeneous teams stay coherent.
- **Anyone-on-the-team is in scope.** The same governance that protects the codebase from drift also lets PMs, designers, and less-code-savvy contributors participate safely. One audience list — role differentiation lives in CODEOWNERS, not in separate curriculum tracks.
- **Local-clone is a feature.** No SaaS to procure, no SSO/RBAC integration, no data leaves the org, every adopting team can fork their own variant. The security and finance objections are answered before they're raised.
- **Complementary to, not competing with, official BMAD.** The portal layers team rituals and governance on top of the framework BMAD already teaches.

## Who This Serves

**Primary — Engineer/Team-Trainee.** Anyone on a team working in a BMAD-enabled repo: engineers, PMs, designers, leads. Mixed coding skill, mixed AI-tool preferences, ~3 hours self-paced or one half-day team workshop.

**Primary — Stakeholder.** Engineering and product leaders evaluating whether to adopt BMAD across their org. Need to see the value in 15 minutes.

**Primary — Facilitator.** Engineering leads who will run internal workshops. Need a guide they can pick up with under two hours of prep.

**The named POC**: one product team, currently with zero BMAD experience, authorized to adopt the portal end-to-end. They are the v1 validation audience — they get BMAD, the team rituals, and the governance pattern simultaneously.

## Success Criteria

Per audience, one concrete signal each:

- **Trainee succeeds when** they complete the capstone and open a real PR — in this repo or their team's — with an attached story file *and* the lead approves it on first review without spec-vs-code corrections (the faithfulness check, not just the artifact check).
- **Stakeholder succeeds when** they can articulate the team-rituals + governance thesis back in their own words after the 15-minute demo (the *portal-attributable* signal — what the demo itself can be held to). **Stretch outcome:** the stakeholder greenlights a BMAD pilot on at least one team within 30 days.
- **Facilitator succeeds when** they run a half-day team workshop with under two hours of prep, using `training/facilitator-guide.md` as-is — validated by at least one dry-run with a facilitator other than the builder before v1 ships.

**POC validation plan**: in the first 30/60/90 days the POC team adopts the portal, capture (a) % of merged PRs with a linked story file, (b) lead reports of CODEOWNERS catching at least one drift in 30 days, (c) at least one merged PR from a non-engineer, (d) actual time-to-completion for the capstone vs. the 60–90 min target. These are the go/no-go inputs for v1.1.

**Macro outcome the portal exists to drive:** more teams adopt BMAD across the organization; spec-driven, AI-assisted development becomes the team norm; AI contributions stop landing in `main` without team alignment.

## Scope

**In scope (v1, everything ships together):**

- Three audience entry points: `training/00-start-here.md` (trainee), `training/stakeholder-demo-script.md` (15-min eval), `training/facilitator-guide.md` (workshop).
- Six lessons: (1) What is BMAD, (2) The artifact chain, (3) Stories as tool-agnostic contract, (4) CODEOWNERS + lead-approval gate (including *what* the lead is reading for at the gate), (5) Working as a team — rituals, async checkpoints, mixed AI tooling, (6) Capstone — running the full cycle.
- Three labs: solo, full-team (synchronous), async-team — practice surface for the rituals.
- Capstone (60–90 min): trainee produces 1 product brief + 1 epic + 2 stories + 1 architecture decision record, end-to-end through BMAD.
- One-page team-rituals checklist for trainees to pin in their own repo as post-capstone reinforcement.
- Per-tool friction notes for Claude Code, GitHub Copilot, OpenAI Codex (CLI + ChatGPT agent), OpenCode — sequestered in `training/tools-reference.md`, dated and versioned with a "verified against versions X" header.
- Working CODEOWNERS, branch protection notes, and PR template committed at `.github/`.
- Local-only distribution: `git clone && npm install && npm run dev`.
- E2E tests so the portal is treated as a real product, not throwaway demo code.

**Explicitly out of scope:**

- Not a BMAD reference manual (official docs cover that).
- Not a tool-specific tutorial — agnostic by design.
- No coverage of agentic tools beyond the four named (Cursor, Aider, Continue, etc., are out).
- Not hosted/SaaS — local-only forever; no SSO, no audit-log SaaS, no central training server.

## Why Now

- **Adoption-governance gap.** 78% of dev teams use AI assistants; only 9% of enterprises are at "Ready" AI governance maturity (Deloitte 2025).
- **Top-cover from authoritative voices.** The April 2026 ACM TechBrief warning gives engineering leaders explicit budget justification for governance training.
- **Tool-agnostic contracts are newly credible.** BMAD v6's cross-platform agent team release (April 2026) makes the mixed-tooling premise practical, not aspirational.
- **The internal moment is right.** A POC team is already authorized; the org wants to get ahead of AI code challenges and align on a unified way of working with AI before the practice ossifies into chaos.

## Vision

In two to three years, BMAD-style spec-driven development is the default mode across the organization. Every team-enabled repo carries the same governance posture: stories are the contract, CODEOWNERS routes review, the lead is the merge gate, and a PM or designer can ship a small change under AI assistance with the same confidence as a senior engineer. **The org's effective contributor base expands without expanding risk** — the governance pattern is what makes that expansion safe.

The spread mechanism is **fork-per-team**: every adopting team forks the portal, tailors CODEOWNERS and rituals to their context, and their fork becomes the next team's reference. No central training function gates throughput; capstone PRs from each team become the visible momentum that triggers the next fork. New AI tools plug in by satisfying the story-as-contract — adopting them is a config change, not a curriculum rewrite.

The portal's role evolves from "introduction" to **the org's reference implementation of how we work with AI** — the canonical artifact a new team forks when it starts a new repo, the answer the Responsible-AI council points at when asked what governance looks like at the code layer.

## Risks & Considerations

| 1 | Single-builder bus factor — one person designing curriculum, building the app, writing artifacts, maintaining tool notes | High | Recruit a co-owner before v1 ships; commit lessons and labs to plain markdown so any teammate can edit |
| 2 | Scope ambition — 6 lessons + 3 labs + capstone + app + E2E + 4-tool notes shipping together is plausibly 3–6 months for one builder | High | Hold v1 scope as defined (deliberate choice — splitting weakens the mixed-tooling demo for the POC); offset with co-owner recruitment (Risk #1), markdown-first lessons, and an explicit timeline acknowledged up front rather than discovered late |
| 3 | Self-reference creates a versioning trap — lesson prose silently rots as PRDs and stories evolve | Medium | Add lesson-to-artifact links as tested references in E2E; quarterly lesson sweep |
| 4 | CODEOWNERS-as-gate concentrates load on the lead engineer; under volume, leads rubber-stamp or become the bottleneck | Medium | Lesson 4 teaches gate-load distribution via multi-owner CODEOWNERS rules; surface the failure mode explicitly |
| 5 | POC-to-team-#2 handoff has no named engine — the org-wide vision lacks a mechanism | Medium | Fork-per-team flywheel: each adopting team forks the portal, customizes CODEOWNERS and rituals to fit their context, and that fork becomes the next team's reference — internal organic spread without a central training team gating throughput. Capstone PRs from the POC team become the visible momentum that triggers fork #2 |
| 6 | Mixed-tool premise weakens if a team is locked to one vendor (common in regulated orgs) | Low | Lesson 5 teaches the story-as-contract abstraction so single-tool teams still benefit; mixed-tool is the headline but not the only path |
