## What went well

**1. Per-story CR cadence held — and counts varied honestly.**
Epic 2 review counts had stabilized at 8–10 (the "always 9 patches" pattern the user called out mid-epic). Epic 3 counts: **9 / 5 / 7 / 5**. The first review (Story 3.1) was substantial because it's the foundation; Stories 3.2/3.4 came in at 5 each because the surface was small and the 14-banned-auth-libs / no-`process.argv` / WCAG / prop-sync issues were genuine. Resisted padding.

**2. The "no auth" property never required a dedicated story.**
Originally I projected a "Story 3.5 — architectural enforcement of no auth" — turned out it was already enforced cross-cuttingly by Story 3.1's two smoke tests (schema contains no `users`/`sessions` after comment-stripping; package.json declares no auth library from a 14-name banned list) plus Story 3.2's import-discipline test plus Story 3.3's client component (no fetch credentials). The architecture's "absence is the design" stance held up at every layer.

**3. E2E DB isolation via `BMAD_DATABASE_PATH` emerged organically.**
Story 3.3 needed e2e to write progress without polluting the dev DB. The env override pattern + Playwright `webServer.env` wiring + Story 3.4's `.sqlite`-extension guard + the tests-pass-explicit-`db`-arg pattern in the storage layer formed a clean isolation seam without invoking heavyweight test-DB infrastructure.

**4. Architecture's "smallest interactive surface" lock prevented feature creep.**
Story 3.3's mark-complete button is one button + an inline `<span role="status">`, not a toast component. AC2 said "toast"; we shipped a smaller surface and documented the deviation in the story file. Saved a component, no UX cost.

**5. `key`-on-parent prop sync was the right pattern.**
First Story 3.3 patch attempt used `useEffect(setState, [prop])` — the lint rule `react-hooks/set-state-in-effect` correctly flagged it. The cleaner React idiom (`key={...}` on the parent forces remount) is what landed. Honest course-correct mid-patch.

## What didn't go well

**1. Story 3.1's review patch broke production.**
The Story 3.1 CR pushed `SCHEMA_PATH` to use `import.meta.dirname` instead of `process.cwd()` for "production-build robustness". Two stories later (Story 3.3) the dev server crashed on every lesson/lab page because Turbopack doesn't populate `import.meta.dirname`. Fixed with a fallback chain. **The over-defensive review patch became a regression.** Lesson: when a review proposes "tighten this for hypothetical production environments we don't actually deploy to", be more skeptical.

**2. React lifecycle issues had no unit-test net.**
Story 3.3 introduced `LessonCompleteButton`. Three real bugs (stale `isSaving` closure → double-click race; useState ignores prop changes; setState after unmount) were caught only at code review, not by unit tests, because the architecture explicitly says "no React-component-level tests at v1". For a single component this was fine; if Epic 4's capstone harness introduces more client components, the test net's blind spot will grow proportionally.

**3. Architecture-doc drifts keep appearing.**
Two drifts surfaced this epic alone: the staleness threshold (`>` vs `≥`, fixed pre-Epic-3) and the reset-progress "hardcoded target" claim (still pending a doc edit). Pattern: implementation honors the spirit, the doc text lags. Fixing in batches isn't working — needs a process change.

**4. `clickAndWaitForPersist` was first written to match any POST — flaky risk.**
Initial Story 3.3 e2e helper waited on any `/api/progress` POST regardless of body. Story 3.3 review caught it; the fix was a body-matching predicate. Same shape as Story 2.2's "lesson 6 negative-Next assertion" review finding — under-specified e2e selectors are a recurring blind spot.

**5. No retro for Epic 2.**
Epic 1 retro happened; Epic 2 (5 stories) never got one. Lessons from Epic 2 (test infrastructure landing pattern, the architecture-vs-epics drift on staleness threshold, the `<Markdown>` typography polish that became R1+R2) are captured in story files but not synthesized. Should have done it.

## Patterns + lessons

| Pattern | Where it surfaced | Take-away |
|---|---|---|
| Review patches can introduce regressions | Story 3.1 → Story 3.3 (Turbopack `import.meta.dirname`) | Be skeptical of "tighten this for hypothetical contexts" patches; require evidence the hypothetical context exists |
| Architecture's "no auth" enforced cross-cuttingly, not as a single story | Stories 3.1+3.2+3.3 all carry pieces | Some properties are negative-space; don't synthesize a phantom story for them |
| E2E DB isolation via env var is the lightest seam | Story 3.3 / Story 3.4 | Pattern works; replicate for any future external-state surface (capstone artifacts, log files) |
| Patch counts vary honestly when scope varies | 9 / 5 / 7 / 5 | The "always 9" worry was unfounded; trust the triage |
| React lifecycle bugs caught at review, not at impl | Story 3.3's three lifecycle patches | Architecture's "no component tests at v1" is correct for the scale today; revisit if client-component count grows |
| Architecture-vs-epics drift surfaces story-by-story | Threshold inequality, reset hardcoded-target | Need a "drift sweep" task in Epic 5 retro pass to reconcile in batch |

## Epic 4 prep

**Epic 4 — Capstone Harness.** Stories (per `epics.md`):
- 4.1: Resolve capstone-session schema shape; extend progress store for `capstone-session` + `capstone-step` kinds
- 4.2: `POST /api/capstone/save` Route Handler + working-tree artifact write
- 4.3: `/capstone` overview route with resume-or-start
- 4.4: `/capstone/[step]` per-step page with form for each artifact type

**Dependencies on Epic 3:**
- ✅ `progress` table accepts `capstone-session` / `capstone-step` kinds at the type layer (Story 3.1's `ProgressKind` already includes them)
- ✅ `upsertProgress` works against any kind (only Zod gates `lesson|lab` at the API boundary)
- ✅ E2E DB isolation works (capstone tests can use the same `BMAD_DATABASE_PATH` seam)
- 🔶 Architecture's flagged "capstone-session schema shape" design point is Story 4.1's load-bearing first decision — the architecture doc says this is the place to resolve it

**Likely friction points (predictions to revisit at Epic 4 retro):**
- Capstone artifact write path (`./_bmad-output/capstone/<utc-timestamp>/`) is on a tree the reset script intentionally does NOT touch (NFR-R3). The capstone Route Handler will need to mkdirp the timestamp dir, write multiple files (brief, epic, story-1, story-2, ADR), and not collide with concurrent capstone runs.
- A second Route Handler (`POST /api/capstone/save`) doubles the API surface. Zod schema reuse / shared error-envelope shape is worth committing to before Story 4.2 ships.
- `/capstone` overview needs to query the most-recent `capstone-session` row by id DESC. Story 3.1's storage layer doesn't have a "list sessions" helper; Story 4.1 will need to extend it.
- The `/capstone/[step]` form is a multi-step capstone harness — likely the second + third client component (after Story 3.3's button). Lifecycle issues like Story 3.3's will recur; B1/B2 action items should be in mind.
- Architecture's `'capstone-session'` semantics: `completed_at IS NULL` while in progress; set when all steps complete. That's a different `completed_at` lifecycle than lessons/labs (where `completed_at` is the terminal state). Story 4.1 needs to spell this out clearly to avoid confusion in the storage layer.

## Closing

Epic 3 was the architecture's first real exercise of the dynamic stack — SQLite, a Route Handler, a client component, a destructive script — and it held up. The "no auth" property held without a dedicated story; the e2e DB isolation pattern emerged cleanly; the `key`-on-parent prop sync replaced an ill-advised `useEffect`. The biggest learn was the Story 3.1 review patch that broke production three stories later — review patches are not free, and "tighten for production" can backfire when the production we actually deploy to has different runtime semantics than the auditor was modeling.

Test net more than doubled (60 → 112 unit; 16 → 20 e2e). React lifecycle blind spot is real but bounded by the architecture's intentional scope. Onward to Epic 4.
